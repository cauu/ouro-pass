# Claude Review — S0001-poolops-issuer-backend

Scope reviewed: case 1 spec-changes (whole server/ at HEAD)
Base/diff: 19da9ae..HEAD -- server/  |  Files: 75 non-test, ~6062 LOC
Spec standard: docs/specs/20260623T0041-S0001-poolops-issuer-backend.md

## Assessment
Overall: REQUEST_CHANGES
The architecture is clean (functional-core rules engine, isolated crypto package, parameterized SQL everywhere, fail-closed wiring) and the COSE/CIP-8 trust-root is correctly bound to both the nonce and the claimed key. But the one-time-use "Consume" primitives that underpin nonce, authorization-code and activation-code replay protection are not atomic on PostgreSQL (the documented prod driver), allowing double-spend; and the epoch reconciler can wedge so that revocations/downgrades never apply. Both must be fixed before merge.

## Findings

### P0 — Critical

- **[server/internal/store/repo_authnonce.go:47-71, server/internal/store/repo_authorizationcode.go:35-57, server/internal/store/repo_activationcode.go:35-61]** One-time "Consume" is a check-then-act race → replay / double-spend on PostgreSQL
  - Issue: All three single-use primitives do `SELECT ... WHERE x=?` → inspect `consumed_at`/`status` in Go → `UPDATE ... SET consumed_at=? WHERE x=?`. The UPDATE has **no `consumed_at IS NULL` guard**. They run inside `store.WithTx`, but `WithTx` (`store.go:84`) calls `BeginTx(ctx, nil)` with engine-default isolation. On PostgreSQL that is READ COMMITTED with no row lock on the `SELECT`, so two concurrent redemptions of the same nonce/code both read it unconsumed, both pass the Go check, and both commit the UPDATE. SQLite happens to be safe only because `Open` forces `SetMaxOpenConns(1)` (`store.go:62`), serializing all writers — so this is invisible in the SQLite test suite.
  - Risk: Defeats replay protection for the entire trust chain: a signed wallet nonce (§4.5) can be consumed twice, an authorization code (§4.3) redeemed for two token pairs, and an activation code (§4.4) redeemed to create two Telegram subscription sessions. Directly violates TC-3/TC-6/TC-9 ("nonce/code consumed exactly once") on the production driver.
  - Suggested fix: Make the write the gate, not the read. Change each UPDATE to `UPDATE ... SET consumed_at=? WHERE code=? AND consumed_at IS NULL` (activation: `... WHERE code=? AND consumed_at IS NULL`) and treat `RowsAffected()==0` as `domain.ErrConsumed`. Keep the prior `SELECT` only to distinguish `ErrNotFound`/`ErrExpired`/`ErrPurpose`. This is dialect-agnostic and removes the window entirely.

### P1 — High

- **[server/internal/worker/reconciliation/reconciliation.go:57-86, 101-108]** A single credential error wedges the whole epoch — revocations/downgrades never apply
  - Issue: `Reconcile` returns on the first `Eligibility` or store error for any one session (lines 59, 63, 75, 82), so one transient chain-query failure for a single credential aborts reconciliation for **all** remaining active sessions. Worse, in `Run` `lastEpoch` is advanced only on a fully-successful pass (line 105), so a credential that fails persistently (e.g. a deregistered/odd account that the chain source errors on) blocks every downgrade and expiry for that epoch indefinitely — members keep tier/access they should have lost.
  - Risk: Defeats C7/TC-11 (downgrade/expire on the epoch boundary). This is a revocation-availability hole: loss-of-access events silently never fire. An attacker who can make their own credential error on the chain source could even keep the whole pool's reconciliation stuck.
  - Suggested fix: Make the per-session loop fault-isolating: log and `continue` on individual `Eligibility`/store errors, count failures into `Result`, and advance `lastEpoch` for the epoch regardless (or only re-process the failed subset next pass). Do not let one bad session gate the others or the epoch cursor.

- **[server/internal/httpapi/handlers_admin.go:138-143]** Admin IP is taken from raw `X-Forwarded-For` (spoofable) → login throttle evasion + audit-log poisoning
  - Issue: `clientIPFromReq` returns the entire `X-Forwarded-For` header verbatim when present, with no trusted-proxy handling. This value feeds admin-login auditing (`Verify(..., clientIPFromReq(r))`, line 59) and the audit IP on every sensitive op (`handlers_admin_resources.go:48`). Meanwhile the public-plane rate limiter keys on `RemoteAddr` only (`middleware.go:120`), so the two layers disagree.
  - Risk: A client can set arbitrary `X-Forwarded-For` to (a) forge the IP recorded in `AdminUser`/`AuditLog` for every admin action (anti-forensics, framing another IP) and (b) rotate the value to evade any future per-IP admin-login throttling. With no proxy in front, `RemoteAddr` is the only trustworthy source and the header should be ignored.
  - Suggested fix: Introduce one trusted-IP resolver used by both middleware and admin handlers: if behind a known proxy, parse the rightmost untrusted hop of `X-Forwarded-For` against a configured trusted-proxy count; otherwise use `net.SplitHostPort(r.RemoteAddr)`. Never store/trust the raw full header.

- **[server/internal/httpapi/middleware/middleware.go:131-190]** Idempotency cache grows unbounded, caches failures, and is not scoped to method/path
  - Issue: (1) `put` (line 184) inserts unconditionally and entries are only evicted lazily when *that same key* is fetched again after TTL (`get`, line 175). A key written once and never re-requested is never freed → the `cache` map grows without bound (memory-exhaustion DoS); there is no sweeper, unlike the IP limiter. (2) `put` is called for any status including 4xx/5xx (line 167), so a client that hits a transient 500 with `Idempotency-Key: X` gets the cached 500 replayed for 10 minutes and can never retry that key successfully. (3) The cache key is the bare header value, shared across `/api/oauth/token` and `/api/activation/create` (both use `idem`, `router.go:58,61`), so the same key reused on two endpoints replays the wrong body.
  - Risk: Unauthenticated memory-exhaustion DoS (the endpoints are pre-auth), plus correctness bugs (poisoned error replay, cross-endpoint body confusion).
  - Suggested fix: Add a background TTL sweeper (or cap with LRU); only cache 2xx responses; scope the key as `r.Method + " " + r.URL.Path + "\x00" + key`.

- **[server/internal/core/oauth/token.go:106-117, 233-239]** Refresh rotation is not atomic — rotate + mint span multiple non-transactional writes
  - Issue: On refresh, `SetStatus(..., GrantRotated)` (line 106) and then `mint` (which calls `issuedTokens.Create` and `grants.Create`, all passing `nil` for the tx — token.go:224/233) execute as independent writes with no shared transaction. A failure or crash after `SetStatus` but before the new grant is persisted leaves the old grant `Rotated` with no successor.
  - Risk: The legitimate user is left with a consumed refresh grant and no replacement → must re-authenticate from scratch (self-inflicted DoS, recoverable). It also widens the theft-detection window: between the two writes the new access token may be issued without the new grant durably recorded. Not data loss, but a correctness/availability defect on the rotation hot path (TC-7).
  - Suggested fix: Wrap the rotate-then-mint sequence in one `store.WithTx` and thread the `*sql.Tx` through `SetStatus`/`issuedTokens.Create`/`grants.Create` (they already accept a `Querier`/tx param). Commit only after the new grant is written.

### P2 — Medium

- **[server/internal/store/repo_refreshgrant.go:97-106]** RevokeChain ignores `rows.Err()` → theft-response revocation can silently truncate
  - Issue: The BFS over `rotated_from` children iterates `rows.Next()` but never checks `rows.Err()` after the loop. A mid-iteration driver/read error exits the loop as if complete, so descendant grants in the rotation chain are left un-revoked.
  - Risk: Under-revocation in the refresh-theft response path (§9.4): an attacker's stolen descendant grants may survive a chain revoke. Low probability but security-relevant.
  - Suggested fix: After the `for rows.Next()` loop add `if err := rows.Err(); err != nil { rows.Close(); return err }` before advancing the BFS.

- **[server/internal/httpapi/router.go:67-68]** Introspection and revocation endpoints are fully unauthenticated
  - Issue: `/api/oauth/introspect` and `/api/oauth/revoke` are in the public verifier group with only rate limiting — no client authentication. Introspect returns `sub`, `tier`, `exp`, `active` for any token presented (introspect.go:52).
  - Risk: Any party who obtains an access token (e.g. from a leaky resource server log) can probe its live status and read the pseudonymous `sub`/`tier` without credentials (RFC 7662 normally requires the caller to authenticate). Revoke being open is lower-risk (you can only revoke a token you already hold), but introspect is an information-disclosure surface.
  - Suggested fix: If the spec intends introspect for confidential resource servers, require client auth (Basic/bearer) on it; otherwise document the open-introspect decision explicitly and minimize the response (drop `sub`/`tier` for unauthenticated callers).

- **[server/internal/httpapi/handlers_admin_resources.go:39-42]** `requireStepUp` dereferences a possibly-nil admin user; step-up applied inconsistently
  - Issue: `requireStepUp` calls `adminFromCtx` and immediately reads `u.OwnerKeyHash` with no nil check. It is normally protected by `requireSession`, but any future re-ordering would panic (caught by `chimw.Recoverer` → opaque 500). Separately, step-up is enforced only on key rotate (line 325); comparably destructive owner/operator ops — `adminRevokeMember` (cascade token/grant/sub revocation), `adminRegisterClient` (issues a client secret), `adminCancelSub` — require no step-up.
  - Risk: Latent 500/panic, and an inconsistent step-up policy for operations of similar blast radius (an operator-role session can revoke any member or cancel any sub with only the session cookie).
  - Suggested fix: Guard `if u == nil { return admin.ErrUnauthorized }`. Decide a step-up policy by blast radius and apply it uniformly (at minimum member-revoke and client registration should match key rotation).

- **[server/internal/worker/push/push.go:101, 122 (push worker)]** Push worker swallows DeliveryLog errors and spends a rate-limit token per retry attempt
  - Issue: `_ = s.deliveries.Append(...)` discards the audit-write error, yet the job is still marked `Done`; and the limiter `Wait` is invoked once per send *attempt* including retries, so a failing recipient consumes multiple tokens.
  - Risk: Silent loss of the delivery-audit record (TC-10 expects a `DeliveryLog`), and degraded effective fan-out rate under failures.
  - Suggested fix: Log/propagate Append failures; key the limiter on recipients (one token per recipient, not per retry); check `ctx.Err()` at the top of the recipient loop for prompt shutdown.

- **[server/cmd/issuer/main.go:125-157]** Graceful shutdown does not wait for background workers
  - Issue: `runNonceGC`, the Telegram worker, and the reconciler are launched as bare goroutines bound to `sigCtx`; on signal, `run()` calls `srv.Shutdown` and returns without joining them. The process can exit mid-`Consume`/`Upsert`/`SendMessage`.
  - Risk: A worker can be killed mid-transaction during shutdown (TC-1 expects clean SIGTERM). Low probability of corruption given each write is its own tx, but in-flight work is abandoned.
  - Suggested fix: Track workers with a `sync.WaitGroup` and `wg.Wait()` (bounded by `ShutdownTimeout`) after `srv.Shutdown`.

- **[server/internal/store/repo_oauthclient.go:73-95]** `OAuthClient.List` skips `Rebind` and discards decode errors
  - Issue: `List` is the only query not wrapped in `r.s.Rebind(...)` — safe today only because it has no placeholders, but a latent break on PG if a `WHERE ?` filter is ever added. It also discards `decodeStrings`/`parseTS` errors with `_` (lines ~92-95), so a corrupt column yields a silently-empty field instead of an error (inconsistent with `Get`).
  - Suggested fix: Wrap the SQL in `Rebind` for consistency and propagate decode errors as `Get` does.

### P3 — Low

- **[server/internal/utils/crypto/cose.go:114-134]** `checkAlg` tolerates a missing/unparseable `alg` label. Not exploitable (the signature is still verified by `ed25519.Verify` against the supplied Ed25519 key, so a non-EdDSA signature simply fails), but for a trust-root primitive it is safer to require `alg == EdDSA` when a protected header is present and reject otherwise. Also consider validating the CIP-8 `address` header field against the signing key when wallets include it.
- **[server/internal/core/rules/engine.go:41, 93-119]** `ruleConfig.RequiredStatus` is parsed but never used in `satisfies`; and because `InputFromSnapshot` always sets `EpochsDelegated = -1` (engine.go:33), the `MinActiveEpochs`/`GraceEpochs` branch (lines 113-118) is dead at runtime — the ≤2-epoch grace handling (C7) is effectively never evaluated at issue/refresh time. Wire a real delegation-age into the snapshot or document that grace is only enforced via reconciliation lag, and either use or drop `RequiredStatus`.
- **[server/internal/worker/telegram/transport_botapi.go:42-61]** `GetUpdates` does not pass `allowed_updates`, so non-message updates (edited messages, channel posts, callback queries) arrive with `From.ID=0`/`Chat.ID=0`/`Text=""` and get a `/help` reply sent to chat "0". Pass `allowed_updates=["message"]` and skip updates with `From.ID==0`/empty text in the worker loop (telegram.go:186).
- **[server/internal/config/config.go:97-112]** `validate()` does not require `OUROPASS_POOL_ID`. With it empty the issuer is `ouropass:` and eligibility silently never matches (fails closed, so not a security hole), but startup should reject an empty pool id rather than booting a service that can never issue. Add a non-empty `PoolID` check.
- **[server/internal/httpapi/handlers_oauth.go:99,111; handlers_wallet.go; handlers_activation.go:37; handlers_admin.go:36,134]** Several 4xx paths reflect `err.Error()` to the client. Today these are domain validation strings, but there is no guarantee a wrapped DB/internal error can't surface on these paths. Map known sentinels to fixed messages and fall back to a generic `"invalid request"` for unrecognized errors (as `serverError` already does for 500s).

## Spec Compliance

- **C1** Go + chi + net/http, no Gin/Echo/Fiber: met — `router.go:12-13` uses chi + chi middleware on `net/http`; no other framework imported.
- **C2** Issuance unified under `POST /api/oauth/token` (authorization_code + refresh_token), no standalone license endpoints: met — `router.go:58`, `token.go:46-55` dispatches both grant types; no license issue/refresh route exists.
- **C3** Private-key isolation (service holds only issuer signing key + bot token; owner key only for admin-login signing, not stored; node socket read-only): met — only `IssuerKey.encrypted_private_key` and `TelegramToken` persist; admin login verifies an owner signature without storing the key (`admin.go:74-116`); chain node access is LSQ read-only by design.
- **C4** Lovelace as numeric(20)/math/big, no int64/JS number: met — amounts are decimal strings end to end (`domain/stakesnapshotcache.go`, stored TEXT) and compared with `big.Int` in `rules/engine.go:100-108`; no int64/float amount path found.
- **C5** Sensitive fields encrypted at rest (private keys, bot token, client_secret): partially-met — issuer private keys are AES-256-GCM encrypted (`keys.go:59`, `fieldcipher.go`); client secrets and opaque tokens are stored as SHA-256 hashes (one-way, acceptable). However the **bot token is not encrypted at rest** — it is env-only (`config.go:78`) and never persisted, so the at-rest requirement is moot for it, but no DB encryption of any "🔒 column" beyond the signing key was exercised. Treat as met-by-design (only the signing key is stored ciphertext); flag that client_secret is hashed not encrypted (fine for verification, not recoverable).
- **C6** Short-lived access + PoP, EdDSA JWS with no cert chain: partially-met — JWS is EdDSA with kid and no cert hash (`jose.go:48-102`); access TTL 24h and activation 30m are within range (`main.go:37`, `activation.go:13`). Public-client device PoP `cnf.jkt` is set at mint (`token.go:211-216`), but DPoP/PoP proof is **not verified on the refresh path** (deferred per D7, `token.go:81-94`) and there is no proof-of-possession check at the resource server in this codebase — binding is recorded but not enforced.
- **C7** Eligibility from active snapshot, recomputed per epoch by reconciliation, ≤2-epoch grace: partially-met — recompute happens (`reconciliation.go`) and at authorize/token time (`oauth.go:155-172`), but (a) the reconciler can wedge so downgrades/expiries never apply (P1 above), and (b) the grace-epoch logic is dead because `EpochsDelegated` is always -1 (`engine.go:33`, P3 above).
- **C8** No Member table; identity = stake_credential_hash; `sub = base32(HMAC(server_salt, hash))`: met — no member table; `DeriveSub` (`hash.go:31-35`) is exactly base32(HMAC-SHA256(salt, hash)); deterministic and salt-gated. Note: server is wired only when the salt is present (`main.go:105`), so an empty/weak salt disables issuance rather than producing guessable subs — good fail-closed behavior.
- **C9** No cold-key license / cert chain / `/api/certs/*` / CRL; JWKS publishes only signing public keys; owner anchored to allowlist (D9): met — no cert/CRL routes; `BuildJWKS` (`jose.go:113-128`) emits only OKP public keys + status; owner role gated by configured `OwnerKeyHash` allowlist (`admin.go:49-52,81`), matching the accepted D9 approximation.
- **C10** `core/rules.Evaluate` pure (no IO/clock/side-effects, priority-sorted, deterministic); `core/keys` not required pure: met — `Evaluate` (`engine.go:59-90`) takes injected snapshot/rules/epoch, sorts stably by (priority desc, rule_id asc), no IO/clock; `keys` is an explicitly stateful shell.

- **TC-1** Build/health/four-plane routing/SIGTERM: partially-met — router exposes all four planes with health 200 (`router.go`); graceful HTTP shutdown present, but workers are not joined on shutdown (P2). Build not run in this review (read-only).
- **TC-2** Dual-stack repos (SQLite + PG, jsonb/TEXT round-trip): partially-met — repos are dialect-portable (RFC3339Nano TEXT times, JSON-in-TEXT, `Rebind`, BLOB/BYTEA split). But the Consume TOCTOU (P0) means correctness **diverges** between engines (safe on SQLite, broken on PG); the test suite appears SQLite-only, so PG behavior is unverified.
- **TC-3** CIP-30 signData verification, tamper rejected: met (with hardening note) — `cose.go` reconstructs the COSE Sig_structure and verifies Ed25519, binding to the nonce (`Verify(pub, []byte(nonce))`) and to the claimed key via the nonce↔key-hash binding (`walletauth.go:71-89`). Tampered nonce/sig is rejected. P3 alg-tolerance noted.
- **TC-4** Access JWS verifiable by independent verifier via JWKS; correct kid (no cert hash); §9.6 JWKS: met — `jose.Verify` validates against the JWKS keyset; `BuildJWKS` is public-keys-only with status, no chain.
- **TC-5** Eligibility engine (min_active_stake/grace/priority) pure & deterministic: partially-met — pure and deterministic (TC met for purity), min_active_stake and priority enforced; grace/min-epochs path is dead (P3) so that sub-rule is untested at runtime.
- **TC-6** Authorize → code; token → access+refresh; ineligible → not_eligible: met — `oauth.go:114-149` and `token.go:121-151`; ineligible returns `ErrNotEligible` mapped to 403/not_eligible (`handlers_oauth.go:108`).
- **TC-7** Refresh rotation + theft revoke chain + downgrade: partially-met — rotation issues new access+refresh and replay of a `Rotated` grant triggers `RevokeChain` (`token.go:69-72`); but rotation is non-atomic (P1) and `RevokeChain` can truncate on a read error (P2).
- **TC-8** Key rotate → JWKS overlap; revoke reflected in introspect: met — `keys.Rotate` demotes actives to rotating and JWKS publishes active+rotating overlap (`keys.go:53-116`); a revoked key drops out of JWKS so its tokens fail verification, and issued-token revoke flips the ledger status checked by `Introspect` (`introspect.go:48`).
- **TC-9** Activation token + deep link; bot `/start <code>` creates session + consumes jti zero-chain-query; `/status`/`/unsubscribe` by from.id: partially-met — flow implemented (`activation.go`, `telegram.go:84-137`) and bind-time eligibility is zero extra chain lookup at delivery; but the activation Consume double-spend (P0) means the "consume exactly once" guarantee fails on PG.
- **TC-10** PushJob → filter by tier/topic/entitlement → sendMessage rate-limited+backoff + DeliveryLog: partially-met — implemented in `push.go`, but DeliveryLog write errors are swallowed and the job still reports Done (P2).
- **TC-11** Reconciliation downgrades/expires at epoch boundary: partially-met — logic exists but the whole-pass abort + non-advancing epoch cursor (P1) can prevent downgrades/expiries from ever applying.
- **TC-12** Admin owner-key login session; RBAC denies; step-up on sensitive ops; AuditLog: partially-met — login/session/RBAC are correct and every admin resource route is role-wrapped behind `requireSession` (`handlers_admin_resources.go:21-34`, no missing-auth route found); audit is written on mutating ops. But step-up is enforced only on key rotation, not on other destructive ops (P2), and the audit IP is spoofable (P1).

- **Scope drift:** None material. Code stays within spec planes; no cold-key/cert-chain code present (C9 honored). `RequiredStatus` rule field (engine.go:41) and the grace-epoch branch are spec-anticipated but currently dead (iteration candidates, not drift).

## Removal / Iteration candidates
- `ruleConfig.RequiredStatus` (engine.go:41) — parsed, never used. Either implement (compare against snapshot account status) or remove.
- Dead grace-epoch branch (engine.go:113-118) — unreachable while `EpochsDelegated` is hard-coded to -1 (engine.go:33). Wire real delegation age or remove and document grace-via-reconciliation.
- `bytesEqual` (cose.go:138-148) is a non-constant-time compare; for the nonce/payload equality it is fine (public value), but if reused for secrets prefer `crypto/subtle.ConstantTimeCompare`. Consider switching to keep the crypto package uniformly constant-time.
- `OAuthClient.List` un-Rebound query (repo_oauthclient.go:73) — fold into the `Rebind` convention.

## Notes / residual risk
- The COSE/CIP-8 trust root is implemented correctly: the public key is supplied by the caller and independently bound to the nonce (the nonce was bound to `blake2b224(vkey)` at challenge time, re-checked at verify, `walletauth.go:77`), and the signature is verified over a hand-assembled `Sig_structure` with `external_aad=[]`. No verification-bypass found. The only hardening is strict alg enforcement (P3).
- SQL injection: none found. Every query is constant SQL with bound parameters; no `fmt.Sprintf`/concatenation builds SQL from user input anywhere in `store/`.
- SSRF on chain adapters is low: the staking-source base URL is operator config, and the stake credential is sent in the POST body (`koios.go:50-53`), not interpolated into the URL/host. No user-controlled host.
- AES-GCM field encryption is correct: 32-byte key enforced, fresh random 12-byte nonce prepended per Seal, tag authenticated on Open (`fieldcipher.go`). Random nonce uniqueness relies on `crypto/rand`; at the expected key-count volume the birthday risk is negligible.
- JWKS leaks no private material (`jose.go:113-128` serializes only public OKP keys). `jose.Verify` validates time claims and binds to the keyset's EdDSA keys; alg is pinned via the JWKS `alg=EdDSA` set on each key (`jose.go:121`).
- Test coverage is meaningful for the pure/unit layers (rules, COSE golden vectors, hash) but the SQLite-only integration posture hides the P0 Consume race and the PG/SQLite isolation divergence (TC-2). Highest-value missing tests: a concurrent double-redeem test against PostgreSQL for nonce/auth-code/activation-code; a reconciler test where one session errors (to catch the whole-pass abort); and a refresh-rotation crash-injection test for atomicity.
- `ActiveSigner` picks the newest active key by `created_at` ordering (`repo_issuerkey.go:43`, `keys.go:94`); correct while a single active key exists, but if two ever coexist with equal second-granularity `created_at` the tiebreak is undefined — minor, acceptable given rotation demotes the prior active.
