> 为已部署用户提供一条命令的**升级**:热备 → (可选)指定版本 → 拉镜像 → 重建 → 健康校验 → 失败给回滚指引。经用户确认:这是**去中心化自托管**场景,**简单可靠优先**,可容忍 `docker compose up -d` 的几秒重启空窗;**不做**零停机(Caddy 重试 / 蓝绿)——因为单体内嵌 worker 无多实例协调(无 SKIP LOCKED/leader;Telegram getUpdates 409;push 双发),两实例并行不安全,且复杂度不值。零应用代码改动。

## 1. Requirement Details

### Background
- 当前升级靠手动 `docker compose pull && up -d`;缺少升级前备份与失败回滚指引。
- 架构约束(已核实):push/reconciliation/telegram worker 无 DB 锁/leader 协调,两实例并行会重复副作用 → 真零停机需改应用(workers-off + 向后兼容迁移),不在本期。
- issuer 已优雅停机(SIGTERM→`srv.Shutdown(ShutdownTimeout)`、worker join)、启动自动迁移、有 `/healthz` HEALTHCHECK;镜像无 v 前缀 tag(`0.x.0`/`latest`)。

### Scope
- 新增 `deploy/update.sh`(POSIX sh,沿用 init.sh 的 ROOT 解析:在安装目录运行)：
  1. **前置**:docker、`docker compose`;`.env`、`docker-compose.yml` 存在;从 `.env` 读 `POSTGRES_USER/DB`、当前 `OUROPASS_TAG`。
  2. **热备**(默认开,`--no-backup` 关):postgres 在跑则 `docker compose exec -T postgres pg_dump` 逻辑备份 → `backups/db-<ts>.sql.gz`,并 `cp .env backups/env-<ts>.bak`(`.env` 含 FIELD_KEY,务必留存);不在跑则告警跳过。
  3. **指定版本**(`--tag X`):改 `.env` 的 `OUROPASS_TAG=X` 再升级;不传则用 `.env` 现值(如 `latest`)。
  4. **拉取 + 重建**:`docker compose pull`(旧容器照常服务)→ `docker compose up -d`(接受几秒重启;旧容器优雅 drain,新容器启动并自动迁移)。
  5. **健康等待**:轮询 issuer 容器 `HEALTHCHECK` 直到 healthy(超时阈值)。
  6. **失败回滚指引**:不健康/超时 → 打印回滚步骤(把 `OUROPASS_TAG` 改回旧值 + `up -d`;必要时从 `backups/db-<ts>.sql.gz` 恢复)并非零退出。
- `set -eu` + 错误 trap;幂等(可重复运行)。
- README/docs 升级章节接入 `update.sh`。

### Constraints
- C1 **零应用代码改动**:仅新增 `deploy/update.sh` + 改 `README.md`/`docs/deployment.md`。
- C2 **简单可靠**:不引入 Caddy 重试 / 蓝绿 / 双实例;容忍短暂停机。
- C3 **数据安全**:升级前热备(一致性逻辑快照 + .env);回滚指引明确;迁移前向(forward-only),回滚镜像后若 schema 已变需从备份恢复(文档警示)。
- C4 **POSIX sh 零硬依赖**:docker/gzip(busybox 即可)之外不强求。

### Non-goals
- 零停机 / 蓝绿 / 滚动(需改应用,见 S0011 讨论;未来如需另立 spec)。
- 自动定时升级、自动回滚执行(仅给指引,不自动 restore)。
- 备份上传远端 / 加密备份(本地 `backups/` 即可,文档提示异地留存)。

## 2. Outline Design
```sh
#!/usr/bin/env sh
set -eu ; ROOT=$(cd "$(dirname "$0")/.." && pwd) ; cd "$ROOT"
flags: --tag X | --no-backup | -h
preflight: docker, docker compose, .env, docker-compose.yml
read .env: PGUSER/PGDB, OLD_TAG=current OUROPASS_TAG
[backup] postgres running? -> mkdir backups; pg_dump | gzip > backups/db-<ts>.sql.gz ; cp .env backups/env-<ts>.bak
[--tag]  set_env OUROPASS_TAG X
docker compose pull
docker compose up -d                 # brief restart accepted
wait_health issuer (poll docker inspect .State.Health.Status, timeout ~90s)
  healthy -> success + version
  else    -> rollback hints (revert OUROPASS_TAG=$OLD_TAG + up -d ; restore db from backup) + exit 1
```
- `set_env`(awk 就地替换/追加)、`on_exit` trap(沿用 init/install 风格)。
- 时间戳用宿主 `date +%Y%m%dT%H%M%S`。

### 2.3 Risk and rollback
- R1 pg 未运行导致备份失败中断升级 → 检测容器状态,未跑则告警跳过(不中断)。
- R2 新版本不健康 → 健康等待超时 → 明确回滚指引(改 tag + up;DB 从备份恢复);不自动改动。
- R3 迁移前向不可逆 → 文档警示:回滚镜像后如 schema 已变,需 restore 备份。
- Rollback:纯新增脚本 + 文档,`git revert` 即可。
